# Project Notes

This document explains how the vector search demo works and what every function in [`app.js`](/vector_search/app.js) does.

## Project purpose

The project is a browser-only vector search demo for science articles.

- Data comes from [`data/articles.json`](/vector_search/data/articles.json).
- The browser loads that JSON file with `fetch()`.
- Each article already has an embedding vector.
- The user query is converted into a vector in the browser.
- The app compares the query vector with article vectors using cosine similarity.
- Results are shown in the UI without any backend or database.

## High-level flow

1. The page loads [`index.html`](/vector_search/index.html).
2. [`app.js`](/vector_search/app.js) runs.
3. `loadDataset()` fetches the JSON file and normalizes all stored article embeddings.
4. The page initially shows all articles.
5. When the user searches, `searchArticles(query)` creates a query embedding and ranks the articles.
6. `renderResults()` draws the best matches on the page.

## Constants and shared objects

### `DATA_URL`

```js
const DATA_URL = "./data/articles.json";
```

This is the path to the JSON dataset file.

### `RESULTS_LIMIT`

```js
const RESULTS_LIMIT = 5;
```

This limits search output to the top 5 ranked matches.

### `THEMES`

`THEMES` defines the embedding space used by the demo.

Each item has:

- a theme name such as `Space` or `Climate`
- a list of keywords that help map the user's query into that theme

This means the query embedding is not generated by a machine learning API. It is built from a simple keyword-based scoring system so the demo stays lightweight and easy to understand.

### `elements`

`elements` stores references to important DOM nodes:

- the search form
- the input field
- the results container
- the results metadata text
- the dataset status message
- the result card template
- the preset query buttons

This avoids repeating `document.querySelector(...)` throughout the file.

### `dataset`

```js
let dataset = [];
```

This variable stores the loaded article data after the JSON file is fetched.

## Function-by-function explanation

### `tokenize(text)`

Purpose:
- Converts a string into clean lowercase tokens for query processing.

What it does:
- converts text to lowercase
- removes punctuation and special characters
- splits the text into words
- removes empty values

Why it exists:
- Search input needs to be standardized before the app can compare it against the theme keywords.

Example:

```js
tokenize("Ocean warming, coral reefs!");
// ["ocean", "warming", "coral", "reefs"]
```

### `normalize(vector)`

Purpose:
- Converts a vector into unit length.

What it does:
- computes the vector magnitude
- if the magnitude is zero, returns a zero vector
- otherwise divides each value by the magnitude

Why it exists:
- Cosine similarity works best when vectors are normalized.
- This prevents larger raw numbers from unfairly dominating comparisons.

### `buildQueryEmbedding(query)`

Purpose:
- Converts the user's text query into an embedding vector that matches the same theme space as the articles.

What it does:
- tokenizes the query
- loops through each theme
- counts how many query words exactly match the theme keywords
- adds a small fallback score when a token partially matches a keyword
- normalizes the final vector

Why it exists:
- The stored articles already have vectors.
- To compare a query with those articles, the query must also become a vector with the same dimensions.

Important detail:
- The output vector length matches the number of themes in `THEMES`.

### `cosineSimilarity(vectorA, vectorB)`

Purpose:
- Measures how similar two vectors are.

What it does:
- multiplies corresponding values from both vectors
- adds them together
- returns the final similarity score

Why it exists:
- Vector search is based on comparing embeddings.
- Higher cosine similarity means the article is closer to the query in vector space.

### `topThemes(vector, limit = 2)`

Purpose:
- Extracts the strongest theme labels from an embedding vector.

What it does:
- pairs each numeric value with its theme name
- removes weak theme scores
- sorts themes from strongest to weakest
- returns the top theme names

Why it exists:
- It makes the UI more understandable by showing labels like `Climate, Ocean` instead of only raw numbers.

### `renderEmptyState(message)`

Purpose:
- Shows a simple message in the results area.

What it does:
- replaces the results container content with one styled empty-state block

Why it exists:
- Used when there are no matching results or when the app needs to show a helpful message.

### `showProtocolWarning()`

Purpose:
- Warns the user when the app is opened incorrectly with `file://`.

What it does:
- updates the dataset status text
- updates the results metadata
- renders a message telling the user to run a local HTTP server

Why it exists:
- Browsers block `fetch()` and some script behavior when opening files directly from disk.
- This function gives a clear explanation instead of leaving the page looking broken.

### `renderResults(query, matches)`

Purpose:
- Displays ranked search results in the UI.

What it does:
- clears the existing results
- if there are no matches, shows a helpful empty state
- updates the results summary text
- clones the HTML result template for each matched article
- fills in category, similarity score, title, summary, source, and theme labels
- appends each result card to the page

Why it exists:
- This is the main renderer for search mode.

Input shape:

```js
[
  {
    article: { ... },
    score: 0.87
  }
]
```

### `renderAllArticles()`

Purpose:
- Displays the entire dataset by default.

What it does:
- creates a list from every article in `dataset`
- clears the current results
- updates the metadata to show that all articles are visible
- renders each article using the same template as search results
- uses `Full dataset` instead of a similarity percentage

Why it exists:
- The app now shows all articles on initial load.
- It is also used when the search box is empty.

### `searchArticles(query)`

Purpose:
- Runs the actual vector search.

What it does:
- builds a query embedding from the input text
- checks whether the query produced any useful signal
- if not, shows a guidance message
- calculates cosine similarity between the query vector and every article vector
- removes zero-score matches
- sorts results by descending score
- limits the output to the top 5
- sends the ranked list to `renderResults()`

Why it exists:
- This is the core search function for the demo.

### `loadDataset()`

Purpose:
- Loads and prepares the article data when the app starts.

What it does:
- checks whether the app is running on `file://`
- if yes, shows the protocol warning and stops
- fetches the JSON dataset
- throws an error if the HTTP response fails
- parses the JSON response
- normalizes every article embedding before storing it
- updates the dataset status text
- calls `renderAllArticles()` so the page starts with visible content

Why it exists:
- The app cannot search until the dataset has been loaded and prepared.

Important note:
- The normalization step makes the stored vectors safer and more consistent for cosine similarity.

## Event listeners

### Search form submit listener

```js
elements.form.addEventListener("submit", ...)
```

Purpose:
- Handles manual user searches.

What it does:
- stops the browser from reloading the page
- reads the current search text
- if the field is empty, shows all articles
- otherwise runs `searchArticles(query)`

### Preset query button listeners

```js
elements.presetButtons.forEach((button) => {
  button.addEventListener("click", ...)
});
```

Purpose:
- Lets users try example searches quickly.

What it does:
- reads the preset query from the button
- fills the search input with that query
- runs the search immediately

## Startup and error handling

### `loadDataset().catch(...)`

Purpose:
- Starts the app and catches load failures.

What it does:
- runs `loadDataset()`
- if something fails, logs the error to the console
- updates the status messages
- shows an empty-state message explaining that the dataset could not be loaded

Why it exists:
- Prevents silent failure during startup.

## Summary

[`app.js`](/vector_search/app.js) is organized around four main responsibilities:

- preparing query text
- comparing vectors
- loading dataset content
- rendering results in the browser

The structure is intentionally simple so the demo is easy to host on GitHub Pages and easy to explain to other developers.
